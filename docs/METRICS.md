# 文件品質指標

> **最後更新**: 2024-11-09
> **負責人**: Documentation Team
> **狀態**: Active

## 總覽

本文件定義用於衡量和追蹤 Enterprise E-Commerce Platform 文件品質、涵蓋率和效能的指標。這些指標指導持續改善並確保文件滿足利害關係人需求。

## 指標理念

- **可衡量**：所有指標有清楚的衡量標準
- **可行動**：指標驅動特定改善行動
- **相關**：指標與文件目標一致
- **平衡**：量化和質化指標的混合
- **透明**：指標對所有利害關係人可見

## 涵蓋率指標

### 文件涵蓋率

**定義**：具有完整文件的系統功能百分比

**衡量方式**：
```
涵蓋率 = (已記錄功能 / 總功能) × 100%
```

**目標**：
- **最低**: 80%
- **目標**: 90%
- **優秀**: 95%+

**追蹤**：
- 自動化 script：`validate-documentation-completeness.py`
- 更新：每日
- 報告：每月

**依類型細分**：

| 文件類型 | 目前 | 目標 | 狀態 |
|---------|------|------|------|
| Viewpoints | 100% | 100% | ✅ 完成 |
| Perspectives | 100% | 100% | ✅ 完成 |
| API Endpoints | 85% | 90% | 🟡 進行中 |
| ADRs | 75% | 80% | 🟡 進行中 |
| Runbooks | 70% | 85% | 🔴 需要改善 |
| 開發指南 | 90% | 90% | ✅ 完成 |

## 品質指標

### 文件準確性

**定義**：準確反映目前系統狀態的文件百分比

**衡量方式**：
- 人工審查抽樣 (每月 10%)
- 使用者回報的不準確性
- 自動化偏離檢測

**目標**：
- **最低**: 95%
- **目標**: 98%
- **優秀**: 99%+

**目前**: 97%

**依類型的準確性**：

| 類型 | 準確性 | 問題 | 狀態 |
|------|--------|------|------|
| 程式碼範例 | 98% | 3 個過期 | ✅ 良好 |
| Architecture 圖表 | 95% | 5 個需要更新 | 🟡 可接受 |
| API 文件 | 99% | 1 個過期 | ✅ 優秀 |
| 配置 | 96% | 4 個過期 | ✅ 良好 |

### 連結健康度

**定義**：有效且運作的連結百分比

**衡量方式**：
```
連結健康度 = (有效連結 / 總連結) × 100%
```

**目標**：
- **最低**: 98%
- **目標**: 99%
- **優秀**: 100%

**目前**: 99.2%

**連結統計**：
- 總連結數：1,247
- 有效連結：1,237
- 損壞連結：10
- 外部連結：342
- 內部連結：905

**依類別的損壞連結**：
- 外部 (已棄用)：6
- 內部 (已移動)：3
- 圖表參考：1

### 拼字與文法

**定義**：每 1000 字的拼字和文法錯誤數量

**衡量方式**：
- 自動拼字檢查：`check-spelling.sh`
- 文法檢查工具
- 人工審查

**目標**：
- **最高**: 每 1000 字 2 個錯誤
- **目標**: 每 1000 字 1 個錯誤
- **優秀**: 每 1000 字 0 個錯誤

**目前**: 每 1000 字 0.8 個錯誤

### 圖表品質

**定義**：符合品質標準的圖表百分比

**品質標準**：
- [ ] 清晰可讀
- [ ] 適當標記
- [ ] 最新且準確
- [ ] 一致的風格
- [ ] 適當的細節層級

**目前**: 94%

**圖表統計**：
- 總圖表數：127
- PlantUML：89
- Mermaid：38
- 需要更新：8

## 新鮮度指標

### 平均文件年齡

**定義**：所有文件自最後更新以來的平均時間

**衡量方式**：
```
平均年齡 = Σ(自最後更新天數) / 總文件數
```

**目標**：
- **最高**: 90 天
- **目標**: 60 天
- **優秀**: 30 天

**目前**: 45 天

**年齡分佈**：

| 年齡範圍 | 數量 | 百分比 |
|---------|------|--------|
| 0-30 天 | 156 | 62% |
| 31-60 天 | 68 | 27% |
| 61-90 天 | 21 | 8% |
| 90+ 天 | 7 | 3% |

### 更新頻率

**定義**：每季度每個文件的平均更新次數

**目前**: 2.3 次更新/文件/季度

**目標**: 2.0+ 次更新/文件/季度

**依類型的更新頻率**：

| 類型 | 更新次數/季度 | 狀態 |
|------|--------------|------|
| API 文件 | 4.2 | ✅ 高 |
| Viewpoints | 1.8 | 🟡 中等 |
| Perspectives | 1.5 | 🟡 中等 |
| ADRs | 0.8 | 🔴 低 |
| Runbooks | 2.1 | ✅ 良好 |

### 更新時間

**定義**：從程式碼變更到文件更新的平均時間

**衡量方式**：
- 追蹤程式碼合併與文件更新之間的時間
- 自動化偏離檢測告警

**目標**：
- **最高**: 7 天
- **目標**: 3 天
- **優秀**: 1 天

**目前**: 2.5 天

**依優先順序的更新時間**：

| 優先順序 | 目標 | 目前 | 狀態 |
|---------|------|------|------|
| 關鍵 | 1 天 | 0.8 天 | ✅ 優秀 |
| 高 | 3 天 | 2.1 天 | ✅ 良好 |
| 中等 | 7 天 | 4.2 天 | ✅ 良好 |
| 低 | 14 天 | 8.5 天 | ✅ 良好 |

## 滿意度指標

### 使用者滿意度分數

**定義**：使用者回饋的平均滿意度評分

**衡量方式**：
- 回饋表單 (1-5 評分)
- 季度調查
- 支援票證分析

**目前**: 4.2/5.0

**目標**: 4.0+/5.0

**依利害關係人的滿意度**：

| 利害關係人 | 分數 | 趨勢 |
|-----------|------|------|
| 開發者 | 4.3 | ↑ |
| 維運 | 4.1 | → |
| 架構師 | 4.5 | ↑ |
| 業務 | 3.9 | ↑ |

### Net Promoter Score (NPS)

**定義**：推薦文件的可能性 (0-10 評分)

**計算方式**：
```
NPS = % Promoters (9-10) - % Detractors (0-6)
```

**目前**: +42

**目標**: +40

**趨勢**: 改善中 (較上季 +5)

### 回饋回應率

**定義**：收到回應的使用者回饋百分比

**目前**: 95%

**目標**: 90%+

**平均回應時間**: 2.1 天

## 效率指標

### 文件速度

**定義**：每個 sprint 建立/更新的文件頁面數

**目前**: 18 頁/sprint

**目標**: 15+ 頁/sprint

**速度趨勢**: 穩定

### 審查週期時間

**定義**：從文件 PR 建立到合併的平均時間

**目前**: 1.8 天

**目標**: 2 天

**細分**：
- 第一次審查時間：0.5 天
- 批准時間：1.2 天
- 合併時間：0.1 天

### 返工率

**定義**：審查後需要大幅返工的文件百分比

**目前**: 12%

**目標**: <15%

**返工原因**：
- 技術不準確：45%
- 資訊不完整：30%
- 風格/格式：15%
- 撰寫不清楚：10%

